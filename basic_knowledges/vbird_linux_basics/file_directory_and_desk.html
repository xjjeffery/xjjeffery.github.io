    Linux 文件、目录和磁盘格式¶

    Linux 的文件权限与目录配置¶

    文件权限¶

    在 Linux 中可以使用 ls -al 命令查看目录下的所有文件属性，如下所示

    图中第一栏的第一个字符表示文件类型，Linux 中文件共有七种文件类型:

    • 常规文件：字符为 -，又可以分为纯文本文件、二进制文件以及数据文件
    • 目录：字符为 d
    • 链接文件：字符为 l，类似 Windows 中的快捷方式
    • 块设备文件：字符为 b，是一些存储数据，以提供系统随机存取的接口设备
    • 字符设备文件：字符为 c，是一些串行端口的接口设备，例如鼠标、键盘等
    • 数据接口文件：字符为 s，这类文件通常被用在网络上的数据交换
    • 数据输送文件：字符为 p，解决多个程序同时读写一个文件所造成的错误问题

    第一栏中的剩余字符以三个为一组，分成三组，分别表示用户所有者，用户组以及其他人的权限，每组的只能依次填 r，w，x 权限，如果没有此权限则用 - 表示：

    • r：可读权限，可以读取和查看文件，可以使用 ls 查看目录内容
    • w：可写权限，可以编辑文件，但是不能删除文件，目录具有可写权限，可以对在目录中进行创建，删除，重命名等操作
    • x：可执行，文件可以被执行，目录具有此权限代表可以进入此目录

    如果需要修改上述的权限，可以使用以下的三个命令：

    • chgrp：修改文件所属用户组，修改的组名必须是在 /etc/group 文件存在的
    • chown：修改文件拥有者，修改的用户名必须是 /etc/passwd 中记录的用户
    • chmod：修改文件的权限，SUID、SGID、SBIT 等权限
      • 使用数字修改文件的权限，如 chmod 777 test.txt
      • 使用符号修改文件的权限，如 chmod u+x g+w o+r test.txt

    目录与文件的权限意义

    权限对文件的意义：

    • r：可读取此文件的实际内容
    • w：可以编辑、新增或是修改该文件的内容(但不能删除该文件)
    • x：该文件具有可以被系统执行的权限

    Linux 中文件是否能被执行，是根据是否具有 x 权限来决定，而跟文件名没有绝对的关系。

    权限对目录的意义：

    • r：表示具有读取目录结构列表的权限，可以查询该目录下的文件名数据，也就是说可以使用 ls 命令将该目录的内容列表显示出来
    • w：表示具有改动该目录结构列表的权限，建立新的文件和目录；删除已经存在的文件和目录(不论该文件的权限是什么)；将已存在的文件或目录进行更名；移动该目录内的文件、目录位置
    • x：表示用户具有能否进入该目录称为工作目录的用途

    第二栏则表示由多少我呢见名链接到此节点，第三栏表示这个文件的拥有者，第四栏表示这个文件的所属用户组，第五栏表示文件的容量大小，第六栏表示文件的创建日期或最近的修改日期，最后一栏为文件名。

    Linux 目录配置¶

    Linux 系统的目录结构是一个目录树架构，在之前也了解过，Linux 的目录默认存在两个目录：. 和 ..，分别表示当前目录和上一级目录，这在相对路径中表示路径名十分有用。Linux 中路径一般有两种表示方式：

    • 绝对路径：从根目录开始写起的文件名或目录名称，如 /etc/passwd
    • 相对路径：相对于目前路径的文件名写法，如 ../home/jack

    Linux 的详细目录配置是依据 FHS(Filesystem Hierarchy Standard)，该标准针对目录树架构仅定义出三个目录下应该放什么：

    • /(根目录)：与启动、还原、系统修复等操作有关，是整个系统最重要的一个目录，所以在进行分区时，此目录所在的分区不要太大，减少发生错误的机会，应用程序所安装的软件最好也不要与此目录放在同一个分区。
    • /usr(unix software resource)：与软件安装/执行有关，在安装时会占用较大的硬盘容量
    • /var(variable)：与系统运行有关，在系统运行后会渐渐占用硬盘容量

    FHS 定义出根目录下应该要有以下这些子目录的存在，如下所示：

    • /bin：存放可执行文件的目录，此目录的命令可以被普通用户使用
    • /sbin：启动过程中所需要的，包括启动、修复、还原系统所需要的命令，这些命令只能由 root 用户来使用，如 fdisk、fsck、mkfs 等
    • /dev：在 Linux 系统上，任何设备与接口设备都是以文件的形式存在于这个目录当中
    • /etc：存放系统主要的配置文件，这些配置文件一般允许普通用户查看，但是只有 root 用户可以修改
    • /lib：存档在启动时会用到的函数库，以及在 /bin 或 /sbin 下面的命令会调用的函数库
    • /boot：放置启动会使用到的文件，包括 Linux 内核文件以及启动选项与启动所需配置文件等
    • /media：放置的是可删除的设备，如软盘、光盘、DVD 等
    • /mnt：放置暂时挂载的设备
    • /opt：第三方辅助软件放置的目录
    • /run：放置系统启动后所产生的各项信息
    • /srv：放置启动的网络服务所需要使用的数据
    • /tmp：让普通用户或正在执行的程序暂时放置文件的地方，这个目录任何人都能够存取，关机后会自动清理
    • /home：系统默认的用户家目录
    • /root：系统管理的家目录
    • /lost+found：这个目录是使用标准的 ext2、ext3、ext4 文件系统格式才会产生的一个目录，目的在于当文件系统发生错误时，将一些遗失的片段放置到这个目录下，不过如果使用的时 xfs 文件系统，就不会存在这个目录
    • /proc：是一个虚拟文件系统，它放置的数据都是在内存当中，不占任何磁盘空间，例如系统内核、进程信息(process)、外接设备的状态及网络状态。比较重要的文件有：/proc/cpuinfo、/proc/dma、/proc/interrupts、/proc/ioports、/proc/net/* 等
    • /sys：与 /proc 目录类似，也是一个虚拟的文件系统，主要记录内核与系统硬件信息相关的内容。
    • /usr：是 Unix Software Resource 的缩写，存放操作系统软件资源，而不是用户的数据，这是一个非常重要的目录，用户的很多应用程序和文件都放在这个目录下，此目录分为以下这些子目录
      • /usr/bin：所有普通用户能使用的命令都在这，/bin 则是使用链接的方式链接至此
      • /usr/lib：基本上与 /lib 相同，/lib 就是链接至此
      • /usr/local：系统管理员在本机安装自己下载的软件，安装到此处以便管理
      • /usr/sbin：非系统正常运行所需要的系统命令，最常见的就是某些网络服务器软件的服务命令，/sbin 是链接至此
      • /usr/share：主要存放只读数据的文件，也包括共享文件，在这个目录下放置的数据几乎是部分硬件架构均可读取的数据，因为几乎都是文本文件，包含 /usr/share/doc —— 软件的说明文档，/usr/share/man —— 在线帮助文件
    • /var：存放经常性变动的文件，包括缓存(cache)、日志文件(log file)以及某些软件运行所产生的文件，包括程序文件，常见的子目录有:
      • /var/cache：应用程序本身运行过程中会产生的一些缓存
      • /var/lib：程序本身执行的过程中，需要使用到的数据文件放置的目录
      • /var/lock：某些设备或是文件资源一次只能被一个应用程序所使用，如果同时有两个程序使用该设备，就可能产生一些错误的状况，因此就得将该设备上锁，以确保该设备指挥给单一软件所使用
      • /var/log：放置日志文件
      • /var/mail：繁殖个人电子邮箱的目录
      • /var/run：与 /run 相同，/run 是链接至此
      • /var/spool：放置一些队列数据，用完后会自动删除

    文件的特殊权限: SUID、SGID、SBIT¶

    SUID¶

    当 s 这个标志出现在文件拥有者的 x 权限上时，就被称为 Set UID，简称 SUID，有以下几个限制：

    • SUID 权限仅对二进制程序有效
    • 执行这对于该程序需要具有 x 的可执行权限
    • 本权限仅在执行该程序的过程中有效
    • 执行则将具有该程序拥有者的权限

    SGID¶

    当 s 这个标志出现在用户组的 x 权限上时，就被称为 Set GID，简称 SGID，有以下几个限制：

    • SGID 对二进制程序有用；
    • 程序执行者对于该程序来说，需具备 x 的权限
    • 执行者在执行的过程中将会获得该程序用户组的支持

    SGID 也可以作用在目录上，当一个目录设置 SGID 的权限后，有以下的限制：

    • 用户若对于此目录具有 r 和 x 的权限时，该用户能够进入此目录
    • 用户在此目录下的有效用户组将会变成该目录的用户组
    • 若用户在此目录下具有 w 的权限(可以新建文件)，则用户在此目录下建立新文件，该新文件的用户组与此目录的用户组相同(协同开发时，就可以实现多人互相修改自己所创建的文件)

    Sticky Bit¶

    Sticky Bit 目前只针对目录有效，对于文件已经没有效果了，是指 t 出现在其他用户的 x 权限上，有以下几个限制：

    • 当用户对于此目录具有 w、x 权限，即具有写入的权限
    • 当用户在该目录下建立文件或目录时，仅有自己与 root 才有权力删除该文件

    Warning

    如果用户对此文件或目录没有可执行权限，却设置了 SUID、SGID、SBIT，那么就会以 S 或 T 来表示，代表空。

    磁盘与文件系统管理¶

    认识 Linux 文件系统¶

    在磁盘上设置一个文件系统之前必须对磁盘分区进行格式化，这因为每种文件系统所设置的文件属性/权限并不相同，为了存放这些文件所需的数据，就需要将分区进行格式化，已成为操作系统能够利用的文件系统格式。

    文件除了文件的实际内容外，还有很多的文件属性，它们是如何存放的呢 —— 在 Linux 系统中，会将文件权限和文件属性存放到 inode 中，而文件的实际数据则存放在数据块中。除此之外，文件系统中还有一个超级区块会记录整个文件系统的整体信息，这三个数据的意义如下：

    • 超级区块：记录文件系统的整体信息，包括 inode 与数据块的总量、使用量、剩余量，以及文件系统的格式与相关信息等
    • inode：记录文件的属性，一个文件占用一个 inode，同时记录此文件的数据所在的区块号码
    • 数据区块：实际记录文件的内容，若文件太大，会占用多个区块

    由于每个文件只会占用一个 inode，并且 inode 中记录数据区块的号码，所以只要能够找打文件的 inode 的话，那么自然就知道存放数据的区块号码，也就能够读取出该文件的实际数据。

    文件系统在一开始就会将 inode 与数据块规划好了，除非重新格式化(或利用 resize2fs 等命令修改其大小)，否则 inode 和数据区块固定后就不再变动。但是如果将 inode 和数据区块都放在一起，在文件系统容量很大时，inode 和数据区块的数量将会很庞大，不容易管理。因此，ext2 文件系统格式化的时候基本上会区分多个区块群组，每个区块群组有独立的 inode、数据区块以及超级区块，如下图所示

    文件系统的前面是一个启动扇区，这个启动扇区可以安装启动引导程序，因此我们可以将不同的引导程序安装在不同的文件系统做前端，而不用覆盖整块磁盘的唯一 MBR，这就能制作多重引导的环境。图中每个区块群组的内容如下：

    • 数据区块：放置文件数据的地方，区块的大小与数量在格式化完就不能再修改(除非重新格式化)；每个区块内最多只能够放置一个文件的数据；如果文件大于区块的大小，则一个文件会占用多个区块，反之即，该区块的剩余容量也不运允许再被使用。在 ext2 文件系统中所支持的区块大小有 1K、2K 以及 4K 三种，不同的区块大小所能支持的最大磁盘容量以及最大单一我呢见容量是不相同的，如下所示：
      • 1KB：最大单一文件的容量为 16GB，最大文件系统的容量为 2TB
      • 2KB：最大单一文件的容量为 256GB，最大文件系统的容量为 8TB
      • 4KB：最大单一文件的容量为 2TB，最大文件系统的容量为 16TB
    • inode table：主要记录文件的属性以及该我呢见实际数据是放置在那几个区块内，基本上有以下这些内容：
      • 该文加你的读写属性
      • 该文件的拥有者与用户组
      • 该文件的大小
      • 该文件建立或状态改变时间
      • 最近一次的读取时间
      • 最近修改的时间
      • 定义文件特性的标识，如 SetUID
      • 该文件真正内容的指向

    ext2 的 inode、数据区块与文件大小的关系

    首先 inode 的数量与大小在格式化时就已经固定了，每个 inode 大小均固定在 128B (新的 ext4 与 xfs 设置为 256B)，每个文件仅会占用一个 inode，因此文件系统能够建立的文件数量与 inode 的数量有关。文件系统读取文件时需要先找到 inode，并分析 inode 所记录的权限与用户是否符合，若符合才能够读取区块的内容。

    inode 只有 128B，那么它是怎么记录一个大文件的信息的 —— 文件系统将 inode 记录区块号码的区域定义为 12 个直接、一个间接、一个双间接与一个三间接记录区，如下图所示:

    通过这种方式可以计算出，ext2 区块格式为 1K 时，能够容纳的最大文件时 16GB，与上述说的是一致的。

    • Superblock：记录整个文件系统相关信息的地方，没有超级区块就没有这个文件系统，区块群组含有超级区块可能是对第一个区块群组内超级区块的备份，可以进行快速恢复，其记录的主要信息有：
      • 数据区块与 inode 的数量
      • 未使用与已使用的 inode 与数据区块数量
      • 数据区块与 inode 的大小
      • 文件系统的挂载时间、最近一次写入数据的时间、最近一次检验磁盘的时间等文件系统的相关数据
      • 一个有效位数值，若此文件系统已被挂载，则有效位为 0，若未被挂载，则有效位为 1
    • Filesystem Description：描述每个区块群组的开始与结束的区块，以及说明每个区段分分别介于哪一个区块之间
    • 区块对照表：通过区块对照表可以知道哪些区块是空的，可以快速找到可使用的空间来处理文件，也可以快速找到原本占用的区块，从而进行释放。
    • inode 对照表：与区块对照表的功能类似，记录使用与未使用的 inode 号码

    Info

    对于 ext 系统超级区块的信息查询，可以使用 dumpe2fs 命令，自己可以根据这个命令的输出结果进行分析。

    与目录树的关系¶

    在初步了解文件系统以后，现在需要考虑一个问题，文件和目录在文件系统中是如何记录数据的呢？

    • 当我们在 Linux 下的文件系统中建立一个目录时，文件系统会分配一个 inode 与至少一个区块给该目录。其中，inode 记录该目录的相关权限与属性，并可记录分配到的那块区块号码，而区块则是记录在这个目录下的文件名与该文件名占用的 inode 号码数据，如下图所示
    • 当我们在 Linux 的 ext2 建立一个一般文件时，ext2 会分配一个 inode 与相对于该文件大小的区块数量给该文件。假设一个区块大小是 4K，建立一个 100KB 的文件，文件系统会分配一个 inode 与 25 个区块来存储该文件

    通过上述的了解，文件名是记录在目录的区块中，所以新增、删除、修改文件名会改变区块中的数据，因此必须需要目录具有 w 权限。要读取某个文件时，就务必会经过目录的 inode 与区块，然后才能够找到那个待读取文件的 inode 号码，最终才能读取到文件的区块中的数据。以读取 /etc/passwd 为例，读取的流程如下：

    • 通过挂载点的信息找到 inode 号码为 XXX 的根目录 inode，且 inode 规范的权限让我们可以读取该区块的内容
    • 经过上一步获取区块的好，在区块中找到内容有 etc/ 目录的 inode 号码
    • 读取 etc/ 的 inode 号码得知用户对此目录的权限，具有 r 和 x 的权限，因此可以读取 etc/ 的区块内容
    • 经过上一步获取区块号码，找到区块中内容时 passwd 文件的 inode 号码
    • 读取 passwd 的 inode 号码得到用户对此文件的权限，具有 r 的权限，因此可以读取 passwd 的区块内容
    • 最后将区块中的数据读取出来

    ext2/ext3/ext4 文件的存取与日志式文件系统的功能¶

    新建一个文件或目录，文件系统的处理方式如下：

    • 先确定用户对于欲新增文件的目录是否具有 w 与 x 的权限，若有的话才能新增
    • 根据 inode 对照表找到没有使用的 inode 号码，并将新文件的权限和属性写入
    • 根据区块对照表找到没有使用的区块号码，并将实际的数据写入区块中，且更新 inode 的区块指向数据
    • 将刚刚写入的 inode 与区块数据同步更新 inode 对照表与区块对照表，并更新超级区块的内容

    如果在最后一步时，因为某些原因导致系统中断，此时就会发生元数据的内容与实际数据存放区产生不一致的情况。这在早期时借由超级区块当中记录的有效位与文件系统状态等来判断是否强制进行数据一致性检查，这种方式的检查很废时。因此就有了日志式文件系统，其方式如下：

    • 预备：当系统要写入一个文件时，会先在日志记录区块中记录某个文件准备要写入的信息
    • 实际写入：开始写入文件的权限和数据，开始更新 metadata 的数据
    • 结束：完成数据与 metadata 的更新后，在日志记录区块当中完成该文件的记录

    在这样的程序中，如果在数据的记录过程中发生问题，系统只要区检查日志记录区块，就可以知道哪个文件发生了问题，针对该问题来做一致性的检查即可，不需要对整个文件系统进行检查。

    文件系统的简单操作¶

    磁盘的整体数据存储在超级区块中，可以使用 df 命令来查看文件系统的整体磁盘使用量，常用的选项有：

    • -a：列出所有的文件系统。包括系统特有的 /proc 等文件系统
    • -k：以 KBytes 的容量显示各文件系统
    • -m：以 MBytes 的容量显示各文件系统
    • -h：以人们较易阅读的 GBytes、MBytes、KBytes 等格式自行显示
    • -H：以 M=1000K 替换 M=1024K 的进位方式
    • -T：连同该硬盘分区的文件系统名称也列出
    • -i：不用磁盘容量，而以 inode 的数量来显示

    查看文件系统的磁盘使用量一般使用 du 命令来查看，常用的选项有：

    • -a：列出所有的文件与目录容量，因为默认仅统计目录下面的文件量
    • -h：以人们易读的容量格式显示
    • -s：仅列出总量，而不列出每个各别的目录占用容量
    • -S：不包括子目录下的总计，与 -s 有点差别
    • -k：以 KBytes 列出容量显示
    • -m：以 MBytes 列出容量显示

    在 Linux 下面的链接文件有两种：一种是类似 Windows 的快捷方式功能的文件，可以让你快速地链接到目标文件，这种称为符号链接；另一种则是通过文件系统的 inode 链接来产生新文件名，而不是产生新文件，这种称为硬链接。

    硬链接：每个文件都会占用一个 inode，文件内容由 inode 的记录指向；想要读取该文件，必须要经过目录记录的文件名来指向正确的 inode 号码才能读取。简单地说，硬链接只是在某个目录下新增一个文件名链接到某已使用的 inode 号码关联记录而已，如下图所示

    如上图所示，将任何一个文件名删除，其实 inode 与区块还是存在的，可以通过另一个文件名来读取到正确的文件数据。并且新增一个硬链接，磁盘的空间与 inode 的数目都不会改变，只是在目录下的区块中多写入一个关联数据而已。

    硬链接还有几个限制：不能跨文件系统、不能链接目录。

    符号链接：建立一个独立的文件，这个文件会让数据的读取指向它链接的哪个文件的文件名。当源文件被删除以后，符号链接就打不开了，符号链接实际找到是原始文件名。如下图所示

    符号链接可以与 Windows 的快捷方式划上等号，并且会占用 inode 与区块，记录的是链接的文件名。

    建立链接的命令是 ln，基本使用语法是 ln [选项] 源文件 目标文件。如果没有任何选项，则会建立的是硬链接，加上 -s 选项就会建立符号链接。

    磁盘的分区、格式化、检验与挂载¶

    要想在系统新增一块磁盘，应该需要操作的有以下几个：

    • 对磁盘进行划分，以建立可用的硬盘分区
    • 对硬盘分区进行格式化，以建立系统可用的文件系统
    • 若想要仔细一点，则可对刚刚建立好的文件系统进行检验
    • 在 Linux 系统上，需要建立挂载点(即目录)，并将它挂载上来

    划分磁盘之前的观察磁盘的分区状态，有以下几个指令可以使用：

    • lsblk：列出系统上的所有磁盘列表
      • -d：仅列出磁盘本身，并不会列出该磁盘的分区数据
      • -f：同时列出该磁盘内的文件系统名称
      • -i：使用 ASCII 的字符输出，不要使用复杂的编码
      • -m：同时输出该设备在 /dev 下面的权限信息
      • -p：列出该设备的完整文件名，而不仅列出最后的名字而已
      • -t：列出该磁盘设备的详细数据，包括磁盘阵列机制，预读写的数据量大小等
    • blkid：列出设备的 UUID 等参数
    • parted：累出磁盘的分区表类型与分区信息

    在之前了解到分区的方式有两种：MBR 和 GPT，所以不同的分区方式使用不同的命令，对于 MBR 的分区方式使用 fdisk 命令，而 GPT 分区方式使用 gdisk 命令，具体的操作可以通过内部的提示获取，两个命令的使用方式基本相同，如下图

    分区设置完毕后并不会立即被更新，需要通过 partprobe -s 来更新磁盘的分区表信息。

    分区建立完毕以后，需要进行文件系统的格式化，格式化的命令是 mkfs，具体的使用语法可以查阅 man 手册。

    建立完文件系统以后就需要进行挂载，挂载前需要确定几个事情：

    • 单一文件系统不应该被重复挂载在不同的挂载点(目录)中
    • 单一目录不应该重复挂载多个文件系统
    • 要作为挂载点的目录，理论上应该都是空目录才行，如果不是空目录，对其进行挂载，则原先的数据会暂时被隐藏，等到此分区被卸载才能正常显示

    挂载的命令是 mount，使用的基本语法为 mount 设备名/UUID 挂载点。卸载的命令则是 umount，使用的基本语法为 umount [选项] 设备名或挂载点，卸载挂载点必须在此文件系统之外。

    上述挂载完以后，在重启系统后还是需要手动重新挂载，并不是一个系统启动后自动挂载的方式。在每次进入 Linux 系统就自动挂载，可以修改 /etc/fstab 文件，修改有一些限制：

    • 根目录是必须挂载的，而且一定要优先于其他挂载点被挂载进来
    • 其他挂载点必须为已建立的目录，可任意指定，但一定要遵守必需的系统目录架构原则
    • 所有挂载点在同一时间之内只能挂载一次
    • 所有硬盘分区在同一时间之内，只能挂载一次
    • 如若进行卸载，必须先将工作目录移到挂载点之外

    文件和文件系统的压缩¶

    常用的压缩命令¶

    在 Linux 中目前有三个常用的压缩命令，分别是：gzip、bizp2 以及 xz，它们的使用方式基本相同，不同点是它们的压缩比，这三个命令的压缩比依次变好，常用的几个命令选项如下:

    • -c：将压缩的数据输出到屏幕上，可以通过数据流重定向来处理，这种处理方式需要自己添加压缩文件名
    • -d：解压缩参数
    • -t：可以用来检验一个压缩文件的一致性，看看文件有无错误
    • -v：可以显示出原文件/压缩文件的压缩比等信息
    • -#：# 为数字的意思，代表压缩等级，-1 最快，但压缩比最差，-9 最慢，但压缩比最好，默认是 -6
    • zcat/zmore/zless：对应于 cat/more/less 来读取纯文本文件被压缩后的压缩文件

    上述三个压缩命令都是只能针对一个文件进行压缩，不能将对一整个目录进行压缩或将一堆文件压缩在一起。在 Linux 中可以使用 tar 命令将多个目录或文件打包成一个大文件，然后再结合上述三个压缩方式进行压缩。

    tar 的基本用法如下:

    • tar [-z | -j | -J] [cv] [-f 压缩后的文件名] 需要压缩的文件名 ...：打包与压缩
    • tar [-z | -j | -J] [tv] [-f 已有的 tar 文件名]：查看文件名
    • tar [-z | -j | -J] [xv] [-f 已有的 tar 文件名] [-C 目录]：解压缩

    tar 命令常用的选项参数有以下几个：

    • -c：建议打包文件，可搭配 -v 来查看过程中被打包的文件名
    • -t：查看打包文件的内容含有哪些文件名，重点再查看文件名
    • -x：解包或解压缩的功能，可以搭配 -C 再特定目录解压
    • -z：通过 gzip 的支持进行压缩和解压缩，此时的文件名最后是 *.tar.gz
    • -j：通过 bzip2 的支持进行压缩和解压缩，此时的文件名最后是 *.tar.bz2
    • -J：通过 xz 的支持进行压缩和解压缩，此时的文件名最后是 *.tar.xz
    • -v：再压缩/解压缩的过程中，将正在处理的文件名显示出来
    • -f filename：-f 后面要立刻接要被处理的文件名，建议将此选项单独写比较好
    • -C 目录：再特定的目录进行解压缩
    • -P：保留绝对路径，即运行备份数据中含有根目录存在
    • -p：保留备份数据原本的权限和属性
    • --exclude=FILE：在压缩过程中，不要将 FILE 打包

    XFS 文件系统的备份与还原¶

    tar 通常是针对目录树系统来进行备份的工作，如果想针对整个文件系统来进行备份与还原，对于 XFS 文件系统推荐使用 xfsdump 和 xfsrestore 两个工具。

    xfsdump 不仅可以进行文件系统的完整备份，还可以进行累积备份，也就是说新备份只会记录与前一次备份的差异，如下图所示

    图中各个 level 的记录文件会放置在 /var/lib/xfsdump/invemtory 中，在使用 xfsdump 的时候需要注意以下几点:

    • xfsdump 不支持没有挂载的文件系统备份，所以只能备份已挂载的
    • xfsdump 必须使用 root 的权限才能操作（涉及文件系统的关系）
    • xfsdump 只能备份 XFS 文件系统
    • xfsdump 备份下来的数据(文件或存储媒介)只能让 xfsrestore 解析
    • xfsdump 是通过文件系统的 UUID 来辨别各备份文件，因此不能备份两个具有相同 UUID 的文件系统

    常用的选项有以下几个，简单的示例如 xfsdump -l 0 -L boot_all -M boot_all -f /srv/boot.dump /boot：

    • -L：xfsdump 会记录每次备份的 session 标头，其后可以填写针对此文件系统的简易说明
    • -M：xfsdump 可以记录存储媒介的标头，其后可以填写此媒介的简易说明
    • -l：指定等级，有 0~9 共 10 个等级(默认为 0，即完整备份)
    • -f：后面接产生的文件
    • -I：从 /var/lib/xfsdump/invemtory 列出目前备份的信息状态

    Note

    xfsdump 默认仅支持文件系统的备份，并不支持特定目录的备份，所以不能使用 xfsdump 备份 /etc 目录。

    xfsdump 的回复只能使用 xfsrestore 这个命令，常用的命令选项如下：

    • -I：跟 xfsdump 相同的输出，可查询备份数据，包括 Label 名称与备份时间等
    • -f：后面接的就是备份文件，企业界很有可能会接 /dev/st0 等磁带机，我们这里接文件名
    • -L：就是 Session 的 Label name，可用 -I 查询到的数据，在这个选项后输入
    • -s：需要接某特定目录，亦即仅复原某一个文件或目录之意
    • -r：如果是用文件来储存备份数据，那这个就不需要使用。如果是一个磁带内有多个文件，需要这东西来达成累积复原
    • -i：进入互动模式，进阶管理员使用的，一般我们不太需要操作它！
